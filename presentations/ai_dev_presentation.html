1. Flash — Introduction

What is Flash?

Flash is a Rust-based domain-specific compiler for the DPU's programmable flow processors (SGE, PTA) and FlexLSO. It compiles .flash programs into JSON artifacts that FunOS loads to program hardware TCAM/instruction memory at runtime.

Pipeline: Source → Scanner (47 tokens) → Parser (recursive descent) → Semantic Model → Encoder → JSON + C bindings

2. Flow Processor TCAM — Structured Match Syntax

The first project was improving how TCAM entries are expressed in Flash. The SGE flow processor has 128 TCAM entries with a 68-bit key — but the flow syntax treated it as opaque hex.

Asking the AI to analyze the problem

Session Snippet — Analyzing TCAM Semantics
Human
Read the Flash compiler source. Look at specifically how the TCAM match semantic is defined and suggest if it is human friendly or improvements can be made?
AI
Current Flash flow syntax: 
flow {
    mask  0xF0000
    match 0x200000
    action do_stuff
}
Problems identified:
  1. Raw hex mask/match is not human-friendly — the 68-bit key is structured (queue type, opcode, error bits), but treated as opaque hex
  2. No key structure declaration — programmer must consult keygen separately to know bit positions
  3. No priority/overlap validation — overlapping patterns not detected
  4. Flow is too low-level — clean handler/flow separation, but flow lacks any abstraction
Suggested: Add key struct declaration + structured match with field-level matching + compiler validations.

The transformation

Before — Raw hex
flow {
    mask  0xf000_0000_0000_0000
    match 0x2000_0000_0000_0000
    action sq_db
}
After — Structured match
key sge_keygen_key @ message {
    msg_type  : 4
    sid       : 1
    cid       : 1
    thd_id    : 2
    queue_type: 2
    ...
}

flow {
    match sge_keygen_key {
        msg_type == MSG_TYPE_SQ_DB
    }
    action sq_db
}

The compiler resolves field names to bit positions, computes mask/value automatically, and validates overlaps and exhaustiveness at compile time.

✅ What worked well
  • AI analyzed 68-bit key structure across compiler + RTL, proposed 4-part improvement
  • Implemented end-to-end: scanner keywords → parser → model → encoder → tests
  • Converted all 17 SGE programs; found a real TCAM collision bug during conversion
⚠️ What needed human push
  • "Use constants for mask and value — it looks ugly" → AI initially generated raw hex in structured syntax
  • Design taste: AI proposes technically correct solutions; human refines ergonomics

3. FlexLSO — From RTL Analysis to Real Programs

FlexLSO Pipeline — Compiler + Programs (New S3 Block)

Inspired by the TCAM work, the next question was: can we extend Flash to handle FlexLSO — a major new S3 hardware block with no prior SW support? This section shows the deliberate approach used to develop it end-to-end.

Phase 1: Start with RTL analysis

Rather than jumping to implementation, the first step was having the AI read and understand the actual hardware — legacy LSO RTL and the new FlexLSO instruction set.

Session Snippet — RTL Deep-Dive
Human
Read the flex_lso_spec and go through the asics RTL repository. Understand ETP FlexLSO specifically focusing on the input WU, NU operate command, LSO operate command, fixed/flex structure that gets fed to FlexLSO instructions. Let me know once done.
AI
ETP FlexLSO: Spec ↔ RTL Mapping

1. Input WU — wudma_nu_com_opr_t. S3 adds two new bits: flex (0=Legacy, 1=Flex) and size (0=32B, 1=64B).

2. LSO Operate Command — Legacy = fixed 128-bit format. Flex = 16B fixed portion (opc, cmd, hdr_mod_ctl, mode, mss, flex_flag, profile, template, proto_ofst) + variable flex data (16B or 48B).

3. Data to FlexLSO engine (128B) — Fixed DS (16B) + ALU State (16B) + Flex DS (up to 48B). Three separate SRAM banks: mem0 (14-bit metadata), mem1 (instructions 0-3), mem2 (instructions 4-7).

This produced a comprehensive 850-line specification extracted from RTL — covering command structures, header stack types, per-packet modification logic, and the 5-stage data pipeline.

Phase 2: Design through Q&A

Before writing any code, the approach was to have the AI propose and then ask clarifying questions. This back-and-forth — 12 clarifying questions across multiple rounds — iterated on key design decisions before a single line of code was written.

Session Snippet — Human-Driven Design Refinement
Human
For header dispatch, I want to pass struct like tcp.seq_num instead of constants like 16. Can you improve on that?
AI
// Before (raw offsets):
compute fix.seq_num + flex.tcp_seq_num_inc
    -> swap header[16, 4]

// After (symbolic):
line inner_l4 -> stack(0) as tcp : tcp_hdr_t {
    compute fix.seq_num + flex.tcp_seq_num_inc
        -> swap tcp.seq_num
}
The compiler resolves tcp.seq_num → offset 16, length 4 from the struct definition. Raw header[ofst, len] still available as fallback.

4. HRW — Closing the Loop with Real Traffic

HRW Pipeline — End-to-End with Datapath Validation

FlexLSO was complete but couldn't be traffic-tested (S3 ETP refmodel not available). The next step was to apply the same approach to HRW (Header Rewrite) on S2, where existing ref model + tests could validate actual datapath behavior.

Design-first approach

Session Snippet — Thorough Design Process
Human
Resume the work on the HRW proposal. Read all the relevant software — Flash compiler, HRW ref model (hrw.cc), FunOS hrw_tests.c. Then, ask me clarifying questions on design.
AI
HRW Flash Flavor — Design Proposal

Pipeline: .flash → structured JSON → FunOS driver parses JSON at runtime → CSR writes → HW/ref model

Key constructs:
• action NAME { ... } — Block of ≤8 HRW-native instructions (movd, movi, addi, etc.)
• profile NAME { step ... } — Sequence of ≤32 ISM steps referencing named actions
• enum hboa_idx { OL_L2=0, ... } — User-defined HBOA layer names
• const CONST_GLOBAL = 0 — Named constant indices

5. Lessons Learned — What Works and What Doesn't

✅ What works well with AI agents
  • Codebase analysis at scale — Reading RTL, compiler internals, thousands of lines of C/Rust and synthesizing understanding
  • Language & compiler design — Propose syntax, iterate on feedback, implement full pipeline
  • Comprehensive test generation — Including negative tests, edge cases, coverage audits
  • Bug detection during migration — Found the csr_oper TCAM collision that humans missed
  • Rapid iteration — Feature request → implementation → tests in minutes, not days
  • Documentation — Specs, user guides, design proposals, session logs
  • Self-review — AI can review its own code and find real issues (overflow, bounds)
⚠️ What needs human direction
  • HW/SW boundary decisions — Which structs are HW-owned vs SW-owned requires domain knowledge
  • Protocol correctness — AI may put RoCE data in the wrong header layer
  • Architecture decisions — When to pivot (FlexLSO → HRW for testability), how to structure files
  • Debugging direction — AI analyzes well but human must point to the right area to investigate
❌ What doesn't work
  • Blind trust on bit-level encoding — Always verify nibble/byte, offset conventions, struct layouts against HW
  • Assuming AI understands HW semantics — Nibble offsets, HBOA-relative addressing were wrong initially
  • Long autonomous runs — Best results with frequent human checkpoints and feedback loops

The key insight

AI as a 10x amplifier, not a replacement

The most effective pattern was human sets direction, AI executes at scale, human reviews and refines. Each Flash area took 1-2 sessions (~4-6 hours) vs what would have been weeks of manual work. The AI handled the tedious parts (parsing all RTL files, writing 200+ tests, implementing compiler pipeline) while the human focused on design decisions, correctness, and ergonomics.
